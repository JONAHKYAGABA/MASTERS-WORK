## 1. The strategy in one paragraph

This is a 22-way multiple-choice classification problem with a closed action space (14 templates parameterized by 3 cells per scenario), small inline data per scenario (under 3,300 tokens), and a brutal time discount in Phase 3 (0% score above 15 minutes per question). The winning approach is a fine-tuned Qwen3.5-35B-A3B that takes the full scenario data inline, calls at most three high-leverage tools (`judge_mainlobe_or_not`, `calculate_overlap_ratio`, `calculate_pathloss`), and answers in one or two model turns. Training data is generated by self-distillation: the same base Qwen, prompted with the ground-truth answer, produces reasoning traces that justify the answer; we filter to keep only traces whose boxed answer matches ground truth, then fine-tune on the filtered corpus with LoRA.

## 5. Stage B: Agent refactor and Phase 1 submission (30 April–4 May)

The provided `main.py` makes the model fetch each data slice via separate tool calls, even though all the data is already inline in `scenario["data"]`. This is the single biggest performance gain we can apply: inline the data, drop the data-retrieval tools, keep only the three high-leverage compute tools.

### 5.1 The new agent flow

```
For each scenario:
  1. Format the inline scenario data (config, user-plane, signaling,
     traffic, MR) into a clean Markdown block.
  2. Build the user prompt: [diagnostic system prompt] + [scenario data]
     + [task description] + [22 candidate options].
  3. Call the model with only 3 tools exposed:
        judge_mainlobe_or_not(time, pci)
        calculate_overlap_ratio(pci_serving, pci_neighbor)
        calculate_pathloss(time, pci)
  4. Allow up to 3 iterations. Each iteration: model either calls a tool
     or emits \boxed{Cx}.
  5. If no \boxed{} after 3 iterations, force a final-answer turn with
     the explicit option list (free_mode behavior, already in the code).
  6. Hard timeout 240 seconds per scenario. On timeout, emit "Insufficient
     data" rather than zeroing out the score.
```

### 5.2 Code changes to `main.py`

1. **Format scenario data inline.** Add a function:

   ```python
   def format_scenario_for_prompt(scenario: dict) -> str:
       """Render scenario data as a single Markdown block."""
       d = scenario["data"]
       parts = [
           "## Network Configuration\n```\n" + d["network_configuration_data"] + "\n```",
           "## User-Plane Time Series\n```\n" + d["user_plane_data"] + "\n```",
       ]
       if d.get("signaling_plane_data"):
           parts.append("## Signaling Plane Events\n```\n" + d["signaling_plane_data"] + "\n```")
       if d.get("traffic_data"):
           parts.append("## Cell-Level Traffic KPIs\n```\n" + d["traffic_data"] + "\n```")
       if d.get("mr_data"):
           parts.append("## Measurement Reports Sample\n```\n" + d["mr_data"] + "\n```")
       return "\n\n".join(parts)
   ```

2. **Restrict tools to 3.** In `AgentsRunner.run()`, after `tool_defs = self.environment.get_tools()`:

   ```python
   ALLOWED_TOOLS = {"judge_mainlobe_or_not", "calculate_overlap_ratio", "calculate_pathloss"}
   tool_defs = [t for t in tool_defs if t["function"]["name"] in ALLOWED_TOOLS]
   ```

3. **Inline scenario into the first user message.** Replace the existing question construction with:

   ```python
   scenario_block = format_scenario_for_prompt(scenario)
   question = f"{scenario_block}\n\n## Task\n{task['description']}\n\n## Options\n{root_causes}"
   messages = [
       {"role": "system", "content": SYSTEM_PROMPT},   # from prompts/system_prompt.md
       {"role": "user", "content": question},
   ]
   ```

4. **Lower max_iterations default to 3.**

5. **Add per-scenario timeout.** Wrap the call:

   ```python
   import concurrent.futures
   with concurrent.futures.ThreadPoolExecutor(max_workers=1) as ex:
       future = ex.submit(self._run_inner, scenario, free_mode)
       try:
           return future.result(timeout=240)
       except concurrent.futures.TimeoutError:
           return self._timeout_fallback(scenario)
   ```

   `_timeout_fallback` returns the catch-all "Insufficient data" option (always present as one of the 22 options).

6. **Fix the result.csv majority-vote bug.** When `num_attempts > 1`, replace `agent_answers[0]` with `Counter(agent_answers).most_common(1)[0][0]`.

7. **Cache `get_tools()` and `get_scenarios()` calls.** Currently re-fetched per scenario. Cache them on the runner.

8. **Add `--quiet` flag** that suppresses tool-call and model-response printing. Use this for production runs.

### 5.3 The system prompt (`prompts/system_prompt.md`)

```
You are a 5G Radio Access Network troubleshooting expert. You are given
a scenario from a vehicle drive test where user throughput dropped
sharply. Your job is to diagnose the root cause and pick the
optimization action that fixes it from the 22 candidate options.

Follow this exact reasoning procedure:

1. SCAN the user-plane time series. Find the timestamp where DL
   Throughput collapses. Note the serving PCI at that moment.

2. CLASSIFY the failure mode by comparing RSRP, SINR, BLER, MCS, RB
   count immediately before and during the throughput drop:
   - RSRP collapses, SINR follows: COVERAGE issue on the serving cell.
     Candidate actions: tilt, azimuth, or transmission power on the
     serving cell.
   - SINR collapses while RSRP holds: INTERFERENCE issue. Look at the
     top-1 neighbor BRSRP at that timestamp; if it is close to or
     above the serving RSRP, that neighbor is the interferer.
     Candidate actions: change azimuth or tilt to reduce overlap, or
     modify PdcchOccupiedSymbolNum on one of the cells.
   - RSRP and SINR both healthy but throughput drops, with low CCE
     allocation success rate or low RB count: SCHEDULER issue.
     Candidate action: PdcchOccupiedSymbolNum to 2SYM, or check test
     server / transmission.
   - High BLER with low MCS: QUALITY issue, treat as interference.

3. CROSS-CHECK the signaling log for the relevant time window:
   - NRRRCReestablishAttempt events: handover failed, mobility issue.
     Look for missing neighbor relationship or A3 offset too high.
   - NREventA3 firing repeatedly with the same neighbor: ping-pong
     candidate. Look at A3 offset.
   - A2 not firing despite low RSRP: A2 threshold too strict, decrease
     CovInterFreqA2RsrpThld.

4. CROSS-CHECK cell-level KPIs in traffic data:
   - High Downlink Weak Coverage Ratio: confirms coverage hole.
   - Low Downlink CCE Allocation Success Rate: confirms PDCCH issue.
   - High Downlink PRB Utilization with throughput drop: load issue.

5. USE TOOLS strategically when they confirm a candidate answer:
   - judge_mainlobe_or_not(t_drop, serving_pci): if True, user is
     outside the mainlobe; this disambiguates azimuth vs tilt.
   - calculate_overlap_ratio(serving_pci, top_neighbor_pci): high
     overlap (>0.3) implicates the neighbor.
   - calculate_pathloss(t_drop, serving_pci): confirms coverage
     degradation.

6. MAP the diagnosis to the parameterized action targeting the correct
   cell. The 22 options always include "Insufficient data" and
   "Check test server and transmission issues" as defensive choices;
   only pick these when the radio-domain signals are clearly clean.

7. EMIT your answer as \boxed{Cx} (single-answer) or \boxed{Cx|Cy|Cz}
   (multi-answer, ascending order, 2 to 4 options).
```

## 6. Stage C: Self-distillation corpus (5–9 May)

We generate the chain-of-thought training data using the base Qwen3.5-35B-A3B itself, conditioned on the ground-truth answer. The model effectively constructs a justification for each correct answer using each scenario's data. We filter aggressively: any trace whose final boxed answer does not match the ground truth is discarded.

### 6.1 Why self-distillation works here

Three things make this approach reliable despite using the base model as its own teacher:

- **Ground-truth conditioning.** We tell the model the answer up front and ask it to produce a reasoning trace that arrives there. Constructing a justification given the answer is a much easier task than diagnosing without it.
- **Hard verification.** We parse the boxed answer from each generated trace. If it does not match the ground truth, we throw the trace away. We never train on a wrong-answer trace.
- **Multi-sample generation.** For each scenario, we generate 3 samples with temperature 0.6. We keep the first one that ends with the correct answer (with structural sanity checks). This trades 3x compute for trace quality.

This is the same pattern as STaR (Self-Taught Reasoner) and ReST: the model generates candidate rationales, a verifier filters them, and the surviving subset becomes the next round of training data.

### 6.2 The teacher prompt template (`traces/teacher_prompt.md`)

```
You are producing training data for a smaller reasoning model.

You are given a 5G drive-test scenario, the 22 candidate optimization
actions, and THE CORRECT ANSWER. Your job is to produce a clean,
step-by-step reasoning trace that arrives at exactly the given answer
using only the data provided.

Constraints:
1. Use ONLY the data provided in the scenario block. Do not invent
   data, cell IDs, or timestamps not present in the scenario.
2. Reference at least one specific timestamp from the user-plane data
   and at least one specific PCI from the network configuration.
3. Follow this diagnostic procedure:
<paste full diagnostic procedure from prompts/system_prompt.md>
4. Length: 200 to 1200 tokens. No padding, no caveats, no apologies.
5. End with \boxed{Cx} for single-answer or \boxed{Cx|Cy|Cz} for
   multi-answer. The boxed value MUST equal the correct answer below.

Correct answer: <GROUND_TRUTH>

Question type: <single-answer | multi-answer>

Scenario data:
<FORMATTED_SCENARIO>

Candidate options:
<22_OPTIONS>

Produce the reasoning trace now.
```

### 6.3 The distillation script (`traces/run_distillation.py`)

```python
# Pseudocode, condensed
import json, os
from openai import OpenAI
from tqdm import tqdm

client = OpenAI(base_url="http://localhost:8000/v1", api_key="dummy")

with open("data/Phase_1/train.json") as f:
    train = json.load(f)

results = []
for scenario in tqdm(train):
    prompt = build_teacher_prompt(scenario)
    accepted = None
    for attempt in range(3):           # up to 3 samples per scenario
        resp = client.chat.completions.create(
            model="Qwen/Qwen3.5-35B-A3B",
            messages=[{"role": "user", "content": prompt}],
            max_tokens=1500,
            temperature=0.6,
        )
        trace = resp.choices[0].message.content
        boxed = extract_boxed(trace)
        if boxed == scenario["answer"] and structural_check(trace, scenario):
            accepted = trace
            break
    if accepted is not None:
        results.append({
            "scenario_id": scenario["scenario_id"],
            "input": format_inference_input(scenario),
            "trace": accepted,
            "answer": scenario["answer"],
            "tag": scenario.get("tag", "single-answer"),
        })

with open("traces/train_traces.jsonl", "w") as f:
    for r in results:
        f.write(json.dumps(r) + "\n")

print(f"Accepted {len(results)} / {len(train)} traces "
      f"({100*len(results)/len(train):.1f}%)")
```

`structural_check` enforces:
- Length 150 to 1500 tokens.
- Trace contains at least one timestamp from `scenario["data"]["user_plane_data"]`.
- Trace contains at least one PCI from `scenario["data"]["network_configuration_data"]`.
- Trace ends with exactly one `\boxed{...}` expression.
- Every 7-digit cell ID mentioned in the trace appears in the scenario.

### 6.7 What if base Qwen produces low-quality traces?

If the layer-1 acceptance rate is below 60% (i.e., fewer than 1200 traces pass after 3 samples each), the base model is struggling to produce ground-truth-conditioned justifications. Mitigations, in order:

1. Increase max_samples_per_scenario from 3 to 5.
2. Lower temperature to 0.4 for more deterministic generation.
3. Add 3 high-quality hand-crafted examples to the teacher prompt as few-shot demonstrations.
4. Last resort: generate a 50-trace bootstrap set manually using ChatGPT or Claude through chat interface (not API), use those as few-shot examples in the teacher prompt, then run distillation. This stays compliant with "Qwen-only in the pipeline" because the generated bootstrap traces are just example formatting, not training data.

## 8. Stage E: Phase 2 submission (17–18 May)

## 10. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `train.json` LFS pull fails | Medium | Blocks everything | Have a Zindi direct-download URL ready as fallback. Day 1 priority. |
| Base Qwen produces low-quality self-distilled traces | Medium | Stage C accuracy drops | Three sample attempts per scenario, structural verification, fallback to manual bootstrap examples (Section 6.7). |
| Fine-tuned model regresses on rare action templates | Medium | Test accuracy drops | Stratified validation split, per-template metrics, reject any checkpoint with template-level regression. |
| Phase 3 organizer GPU is slower than our test H100 | Low | Time discount kicks in | Profile aggressively. Keep wall-clock at 60s/scenario or less so there is headroom on a slower GPU. |
| Phase 2 submission script crashes mid-run | Medium | Wastes one of three submissions | Test the submission script end-to-end on the day before. Reserve Submission 3 as insurance. |
| LoRA adapter fails to load in vLLM on the organizer's instance | Medium | Phase 3 zero score | Smoke test on a fresh instance is mandatory. Test the exact load-adapter command from the README in the zip. |
| Locked file accidentally modified | Low | Disqualification or run failure | `chmod 444` plus CI hash check. Pre-commit hook. |
| GPU spend overruns budget | Medium | Project stalls | Daily cost check. Use spot instances during distillation, on-demand only for the final submission run. Hard cap $700 total. |
| Multi-account or code-sharing accusation | Low | Disqualification | All work in one Zindi account. Repo private. No public discussion of the solution before challenge close. |
